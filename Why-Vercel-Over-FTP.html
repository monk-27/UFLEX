Slide 01 — Current Setup

Understanding the Existing Pipeline

📌 How the Current Deployment Works

The existing setup uses a GitHub → FTP → cPanel pipeline: code is pushed to GitHub, which automatically triggers an FTP-based deployment to the cPanel server. This was a standard and reasonable approach at the time it was set up. However, FTP as a protocol has fundamental limitations that no configuration can fix — it was built in 1971, before modern internet security standards existed.

🔓

FTP Was Designed Without Encryption

FTP sends credentials and file data as plain text over the network. This is a design limitation of the protocol itself — not a configuration issue. Any network observer can read the traffic.

🌐

GitHub → FTP Still Uses Unencrypted Channel

Even when FTP is automated via GitHub Actions, the FTP password must travel over FTP's unencrypted channel every time a deployment runs. The automation does not make the channel secure.

🕵️

No Visibility Into File Changes

cPanel does not track which files changed, when, or by which process. There is no alert system if unexpected changes occur. Issues can go unnoticed for extended periods.

🎯

FTP Ports Are Publicly Known Targets

Automated bots continuously scan the internet for open FTP ports (port 21). Every server running FTP is a well-known target — this is a protocol-level exposure that affects all FTP users globally.

⚡ The Core Issue in One Line

FTP was not designed for modern security requirements. The solution is not to fix FTP — it is to replace it with a modern deployment pipeline. Vercel provides exactly this, using the same GitHub workflow the team already uses, just with a fully encrypted and automated path.

Slide 05 — Security Analysis

FTP Security Risks vs Vercel Security Model

🔴 FTP + cPanel — Known Risks

  • Credentials transmitted as plain text
  • cPanel login panel publicly accessible
  • Shared server environments
  • No built-in Web Application Firewall
  • FTP port 21 actively scanned by bots globally
  • No file integrity monitoring
  • Manual PHP and server patch management
  • No deployment audit log

🟢 Vercel — Security Model

  • No FTP — all transfers via encrypted GitHub integration
  • No direct server access needed by developers
  • Isolated deployment environments
  • DDoS protection built-in at network edge
  • Multi-factor authentication supported
  • Immutable deployments — files cannot be modified post-deploy
  • Automatic security patch updates
  • SOC 2 Type II certified infrastructure
